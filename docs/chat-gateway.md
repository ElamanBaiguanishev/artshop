# chat-gateway

Сервис-шлюз между мессенджерами и остальной системой. Единственный компонент,
который знает про Telegram, WhatsApp и что бы там ни появилось дальше.

## Зачем отдельный сервис

Разделение проведено по способу выполнения, а не ради слова «микросервис»:

- **другой рантайм-профиль.** Вебхуки от провайдеров приходят непредсказуемо,
  требуют мгновенного ответа и ретраятся при ошибке. Это не тот же профиль нагрузки,
  что у CRUD-запросов админки;
- **своя частота релизов.** Провайдеры меняют API независимо от магазина; правка
  под очередное изменение Telegram не должна требовать выкатки витрины;
- **изоляция отказа.** Недоступность мессенджера не должна влиять на приём заказов;
- **переиспользуемость.** Шлюз не знает про магазин и может быть применён в другом
  проекте без изменений.

## Границы ответственности

| Что делает шлюз | Что делает `api` |
|---|---|
| принимает вебхуки провайдеров | владеет диалогами и сообщениями в БД |
| нормализует входящее сообщение в общий формат | связывает диалог с заказом и покупателем |
| отправляет исходящее в нужный провайдер | решает, что и когда отправить |
| дедуплицирует входящие | хранит вложения в S3 |
| отслеживает статусы доставки | показывает переписку в админке |

Формулировка границы: **шлюз занимается транспортом, `api` - смыслом.** Шлюз
не знает, что такое заказ; `api` не знает, что такое `chat_id`.

Вложения: шлюз скачивает файл у провайдера и кладёт в S3, наружу отдаёт ключ.
Так `api` не зависит от того, как конкретный мессенджер отдаёт медиа.

## Интерфейс провайдера

Проектируется сразу под несколько мессенджеров - основание для этого есть
(опыт Telegram, WhatsApp и Max в проде), поэтому места расхождения известны заранее.
Реализация на старте одна - Telegram.

```ts
interface ChatProvider {
  readonly kind: 'telegram' | 'whatsapp';

  // разбор входящего вебхука в нормализованный вид
  parseWebhook(raw: unknown): IncomingMessage[];

  // отправка; идемпотентна по clientMessageId
  send(msg: OutgoingMessage): Promise<SendResult>;

  // ссылка, по которой покупатель начинает диалог
  buildDeepLink(payload: string): string;

  // скачивание вложения по идентификатору провайдера
  fetchAttachment(ref: string): Promise<AttachmentStream>;
}

interface IncomingMessage {
  providerKind: string;
  providerMessageId: string;      // ключ дедупликации
  providerChatId: string;
  providerUserId: string;
  text?: string;
  attachments: AttachmentRef[];
  startPayload?: string;          // payload из deep link при первом запуске
  sentAt: Date;
}

interface OutgoingMessage {
  conversationId: string;
  clientMessageId: string;        // наш идентификатор, ключ идемпотентности
  text?: string;
  attachments?: OutgoingAttachment[];
}
```

Различия провайдеров, ради которых интерфейс именно такой:

- **идентификаторы сообщений** у всех свои, и не у всех они есть на исходящих;
- **медиа** отдаются по-разному: где-то ссылка, где-то идентификатор файла,
  который надо отдельно резолвить;
- **статусы доставки** есть не везде, и приходят они разными способами;
- **инициатива диалога** ограничена по-разному (см. ниже).

## Ограничения Telegram Bot API, влияющие на продукт

1. **Бот не может написать первым.** Пока пользователь не выполнил `/start`,
   отправить ему ничего нельзя. Отсюда поток:

   ```
   сайт: кнопка «Обсудить в Telegram»
     -> t.me/<bot>?start=ord_A17
     -> пользователь жмёт Start
     -> приходит /start с payload ord_A17
     -> шлюз отдаёт payload в api
     -> api связывает диалог с заказом A17
     -> дальше переписка и статусы идут свободно
   ```

   Payload одноразовый и подписанный, чтобы по чужой ссылке нельзя было
   привязаться к чужому заказу.

2. **Вебхук, а не long polling** в проде: дешевле и надёжнее. Локально - polling
   под флагом или туннель.

3. **Скачивание файлов ограничено 20 МБ.** Для фото-референсов достаточно, но случай
   «прислали видео» обрабатывается явным сообщением, а не молчаливой ошибкой.

4. **Получатель уведомлений тоже должен нажать Start.** `chat_id` владелицы
   и менеджеров сохраняется при первом запуске бота.

5. **Лимиты частоты**: примерно тридцать сообщений в секунду суммарно и около одного
   в секунду в конкретный чат. При рассылке статусов это существенно, поэтому
   исходящие идут через очередь с ограничителем, а не напрямую.

## Надёжность

**Входящие.** Вебхук подтверждается сразу, обработка - асинхронно. Дедупликация
по паре `(providerKind, providerMessageId)` с уникальным индексом: провайдеры
повторяют доставку при любой заминке, и без этого переписка задваивается.

**Исходящие.** Отправка идёт из очереди, идемпотентно по `clientMessageId`.
Ретраи с экспоненциальной задержкой, разделение ошибок на временные (сеть, лимит)
и постоянные (бот заблокирован пользователем - помечаем диалог и прекращаем попытки).

**Секреты.** Токен бота только в переменных окружения. Вебхук защищён секретным
путём и заголовком, который Telegram присылает обратно.

## Модель данных

Живёт в основной БД, владеет ею `api`.

### conversations

`id`, `provider_kind`, `provider_chat_id`, `provider_user_id`, `customer_name`,
`order_id` (nullable), `status` (`active` / `blocked` / `archived`),
`last_message_at`, `unread_count`.

Уникальный индекс на `(provider_kind, provider_chat_id)`.

### messages

`id`, `conversation_id`, `direction` (`in` / `out`), `provider_message_id`,
`client_message_id`, `text`, `attachments` (jsonb со ссылками на S3),
`delivery_status` (`pending` / `sent` / `delivered` / `failed`), `sent_at`, `author`
(`customer` / `manager` / `system`).

Уникальные индексы: `(provider_kind, provider_message_id)` для входящих,
`client_message_id` для исходящих.

### chat_links

Одноразовые payload'ы deep link: `payload`, `order_id`, `expires_at`, `used_at`.

## Inbox в админке

- список диалогов с последним сообщением и непрочитанными;
- переписка, ответ прямо из админки, отправка фото;
- **карточка заказа рядом с перепиской** - главная ценность связки: менеджер видит,
  о чём договорились, не переключая вкладки;
- быстрые действия из чата: сменить статус заказа, отправить реквизиты, отправить трек;
- шаблоны частых ответов;
- живое обновление по WebSocket.

## Telegram Mini App

Отдельная поверхность поверх того же бота. Смысл не в копии сайта, а в том,
чего сайт не может:

- заказ **без выхода из переписки**, где человек уже общается с автором;
- авторизация без регистрации через `initData` - валидация подписи выполняется
  на сервере по секрету бота, доверять данным с клиента нельзя;
- статусы заказа и уведомления прямо в Telegram, без писем;
- повторные покупки: у постоянных покупателей бот уже есть.

Переиспользует компоненты фронта, отдельная тема оформления под Telegram.
Делается **после запуска сайта**, не раньше.

## Что осознанно не делается

- собственный протокол реального времени между шлюзом и `api` - хватает HTTP
  и очереди;
- хранение переписки внутри шлюза - он транспорт, состояние принадлежит `api`;
- поддержка провайдера, которого ещё нет в планах: интерфейс готов, реализация
  добавляется по факту.
