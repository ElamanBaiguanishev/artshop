# План реализации

Принцип: **сначала тонкий вертикальный срез до живого сайта, потом наращивание.**
Не строить весь бэкенд, чтобы через месяц начать фронт - иначе нет видимого результата
и проект умирает.

Оценка исходит из режима «вечера и выходные», примерно 8-10 часов в неделю.

| Этап | Что даёт | Срок |
|---|---|---|
| 1. Минимум | сайт, который можно показать и который принимает заказы | 3-4 недели -> **запуск** |
| 2. Чат-шлюз | двусторонняя переписка в Telegram, единый inbox в админке | +2-3 недели |
| 3. Продающая обвязка | фиды, трекинг конверсий, отзывы, английская версия | +3 недели |
| 4. Telegram Mini App | каталог и заказ прямо в мессенджере, пуши статусов | +2-3 недели |
| 5. Второй провайдер и роли | WhatsApp через тот же интерфейс, менеджеры | +2 недели |
| Максимум | платежи, доставка по API, эксплуатация | выборочно |

**Главное правило: каждый следующий этап падает на уже работающий сайт.**
Самый вероятный способ провалить этот проект - не «сделать плохо», а «через три месяца
всё ещё не запущено, азарт кончился, продажи по-прежнему идут через директ».

Важная деталь связности: односторонние уведомления в Telegram из этапа 1 - это уже
зародыш `chat-gateway`. На этапе 2 он не пишется с нуля, а достраивается до двусторонней
переписки. Ничего не выбрасывается.

# МИНИМУМ

Цель: сестра кидает ссылку в Instagram, человек смотрит работы, оставляет заявку,
она получает уведомление в Telegram. Всё.

## Этап 0. Каркас (2-3 вечера)

- монорепо pnpm workspaces: `apps/web`, `apps/api`, `apps/worker`, `packages/shared`
- `docker-compose.yml`: postgres, redis, minio (S3 локально)
- Drizzle + первая миграция, `.env.example`, Biome
- health-эндпоинты, скелет Next-приложения

**Готово когда:** `docker compose up` поднимает инфру, Next отдаёт пустую главную,
API отвечает на `/health`.

## Этап 1. Каталог (неделя 1)

- модель `products` + `product_images` (см. data-model.md)
- админский CRUD товара: создать, отредактировать, опубликовать, снять
- загрузка изображений в S3, порядок перетаскиванием
- публичный каталог: сетка карточек, фильтр по типу работы
- карточка товара: галерея, зум, размеры, материалы, цена
- статусы товара: черновик / в наличии / продано / под заказ
- **проданные не удаляются**, остаются в каталоге с плашкой - это соцдоказательство и SEO

**Готово когда:** сестра добавила три реальные работы, они видны на сайте.

## Этап 2. Обработка медиа (неделя 2, первая половина)

- воркер на BullMQ: приём оригинала -> генерация вариантов (thumb / card / full)
- WebP + AVIF, водяной знак на полноразмерных
- прогрессивная отдача: blurhash или LQIP-заглушка, пока грузится основное фото
- статус обработки виден в админке

**Почему отдельным этапом:** фото с телефона весят по 8 МБ. Без этого мобильная
загрузка убьёт конверсию, а весь смысл витрины - в фотографиях.

**Готово когда:** загруженное с телефона фото в 8 МБ отдаётся на сайте как 120 КБ
и появляется мгновенно.

## Этап 3. Заказы (неделя 2, вторая половина - неделя 3)

- форма заявки на карточке товара: имя, способ связи, комментарий
- кнопка «Написать в Telegram» с подставленным номером работы
- сущность `orders` со статусами и историей переходов
- **уведомление в Telegram** владелице и разработчику при новой заявке
- админка заказов: канбан, смена статуса перетаскиванием, заметки по заказу
- **создание заказа вручную из админки** - см. ниже, это не опциональная фича
- резерв товара при заявке: `reserved` + срок брони, чтобы двое не заказали одну картину

> **Заказ первичен, переписка вторична.**
> Значительная часть разговоров идёт в личных аккаунтах - Instagram Direct, личный
> WhatsApp, личный Telegram. Бот их не видит и не должен: это личная переписка,
> а не рабочий ящик. Лезть туда пользовательским клиентом (MTProto) отвергнуто
> сознательно - и по риску для аккаунта, и по приватности.
>
> Отсюда: заказ должен создаваться **вручную** - поговорили где угодно, зашли
> в админку, выбрали работу, вписали контакт. Диалог привязывается к заказу
> опционально, если он вообще существует. Без этого возникает теневой поток заказов
> в переписке и голове владелицы, и система перестаёт быть источником правды.
>
> Управляемые каналы наращиваются отдельно: бот сейчас, рабочий (не личный) номер
> WhatsApp Business - на этапе 5.

**Готово когда:** заявка с сайта прилетает в Telegram за секунды, товар помечается
забронированным, и заказ по разговору из Instagram заводится в админке за полминуты.

## Этап 4. Обвязка и запуск (неделя 3-4)

- страница «Об авторе» с фото и историей
- редактируемые тексты страниц из админки
- OG-теги: ссылка в Instagram и WhatsApp разворачивается красивой карточкой
  (мелочь, напрямую влияющая на переходы)
- ЧПУ-адреса, `sitemap.xml`, `robots.txt`
- Яндекс.Метрика / GA
- JWT-вход в админку, роли: владелец и админ
- домен, HTTPS, деплой на Dokploy
- мобильная вёрстка вычищена

**Готово когда:** сайт открыт, домен работает, сестра ведёт каталог сама.

> ### Точка запуска
> Дальше **сначала эксплуатация, потом разработка**. Две-три недели реального трафика
> покажут, что действительно нужно. Всё, что ниже, приоритизируется по этим данным,
> а не по нашим догадкам.

# ЭТАП 2. ЧАТ-ШЛЮЗ И INBOX

Детали интерфейса и ограничений - в [chat-gateway.md](chat-gateway.md).

- выделение `chat-gateway` в отдельный сервис из кода уведомлений этапа 1
- интерфейс `ChatProvider`, реализация одна - Telegram Bot API
- приём вебхуков, дедупликация входящих, скачивание вложений в S3
- deep link `?start=<payload>` со связыванием диалога с заказом
  (бот не может написать первым - без этого потока переписка не начнётся)
- исходящие через очередь с ограничителем частоты, идемпотентность по `clientMessageId`
- **inbox в админке**: список диалогов, переписка, ответ, отправка фото
- **карточка заказа рядом с перепиской** - главная ценность связки
- быстрые действия из чата: сменить статус, отправить реквизиты, отправить трек
- шаблоны частых ответов
- живое обновление по WebSocket

**Готово когда:** покупатель пишет боту, сообщение появляется в админке, ответ из админки
доходит в Telegram, переписка привязана к заказу.

# СРЕДНИЙ СЛОЙ (ЭТАП 3)

То, ради чего всё затевалось: превращение витрины в инструмент продаж.

## Фиды товаров

- генератор `products.xml` в формате Meta / Google Shopping
- расписание пересборки, инвалидация при изменении товара
- один фид -> Meta (Instagram, Facebook), Pinterest, Google Merchant Center
- **Pinterest не пропустить**: для картин и декора он часто даёт больше Instagram,
  пины живут годами, аудитория западная - стыкуется с отправками в Европу

## Трекинг конверсий

- Meta Pixel на фронте: просмотр товара, начало заявки, отправка заявки
- **Conversions API**: те же события с сервера, через очередь, с ретраями
- дедупликация по общему `event_id`, генерируемому на клиенте
- та же схема переиспользуется для Pinterest Tag и Google

## Продающие элементы

- отзывы с фотографиями покупателей (заводятся вручную из переписок)
- блок «процесс работы»: таймлапс, фото мастерской, «эта работа заняла 12 часов»
- счётчик проданных работ, лист ожидания на распроданное
- похожие работы в карточке

## Кастом-заказ

- отдельная форма: тип изделия, параметры (имя, цвет, размер), загрузка референса
- отличается от заявки на готовую работу: нет товара, есть спецификация
- сроки и этапы изготовления

## Трекинг заказа покупателем

- страница статуса по секретной ссылке, без регистрации
- ссылка уходит в письме и в мессенджере
- избавляет от потока «ну как там?» в личку

## Интернационализация

- русский и английский: интерфейс, тексты страниц, описания товаров
- мультивалютность: KZT / RUB / EUR
- деньги целыми числами в минорных единицах, курс фиксируется в момент заказа
- курсы обновляются по расписанию

## Уведомления

- письма покупателю: заявка принята, цена согласована, отправлено
- шаблоны писем, редактируемые без деплоя

# ЭТАП 4. TELEGRAM MINI APP

Смысл не в копии сайта, а в том, чего сайт не умеет.

- каталог и карточка работы внутри мессенджера, переиспользуя компоненты фронта
- заказ **без выхода из переписки**, где человек уже общается с автором
- авторизация без регистрации через `initData`; **подпись проверяется на сервере**
  по секрету бота - данным с клиента доверять нельзя
- статусы заказа и уведомления прямо в Telegram, без писем
- тема оформления под Telegram, поддержка светлой и тёмной схемы клиента

**Готово когда:** заказ можно оформить целиком внутри Telegram, не открывая браузер.

# ЭТАП 5. ВТОРОЙ ПРОВАЙДЕР И РОЛИ

- реализация `ChatProvider` для WhatsApp - проверка того, что шов рабочий,
  а не декоративный
- диалоги из разных мессенджеров в одном inbox
- роли `owner` / `manager`: приглашение сотрудника, ограничение прав
  (менеджер не меняет цены и не удаляет работы)
- журнал действий администраторов

Модель ролей и проверки прав на бэкенде закладываются **с первого этапа** - это дёшево
в схеме и дорого прикручивать к готовому приложению. Здесь появляется только интерфейс
управления.

**Готово когда:** сообщение из WhatsApp попадает в тот же inbox без изменений в `api`.

# МАКСИМУМ

Делать **выборочно**, по запросу реальной эксплуатации, а не списком.

## Платежи

- абстракция `PaymentProvider`, первая реализация - ручной инвойс (уже есть по факту)
- локальный шлюз для СНГ (Kaspi или аналог), вебхуки, идемпотентность повторных
  уведомлений, сверка статусов
- для Европы - выставление счёта вручную, пока нет доступного провайдера
- фискализация, если потребуется по юрисдикции

## Доставка

- зоны и правила: город, страна, международная
- расчёт по API перевозчика с учётом габаритов и веса
- трек-номер в заказе, отображение на странице статуса
- таможенные данные: вес, содержимое, товарная позиция
  (у оригинальных картин она отличается от сувениров), подготовка данных для CN23
- честное предупреждение о местных пошлинах и НДС при доставке в ЕС

## Эксплуатация

- Prometheus + Grafana: метрики API, длина очередей, время обработки медиа
- алерты: сайт лежит, воркер встал, фид не собрался
- аптайм-мониторинг снаружи
- CI/CD на GitHub Actions: lint -> test -> build -> deploy
- бэкапы БД и хранилища, проверка восстановления

## Аналитика продукта

- собственная воронка: просмотр -> открытие карточки -> начало заявки -> отправка
- отчёт «смотрят, но не заказывают» - главный источник гипотез
- источники трафика в разрезе работ

## Прочее

- PWA, установка иконкой на телефон
- предзаказ и лист ожидания
- промокоды
- печать этикеток и упаковочных листов
